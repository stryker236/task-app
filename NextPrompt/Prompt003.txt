  1. Criar um logger único
     Implementar um módulo central, por exemplo backend/logger.ts, em vez de console.log espalhado.

     Campos base em todos os logs:
      - timestamp
      - level
      - event
      - requestId
      - user/session/client
      - route
      - durationMs
      - metadata

     Biblioteca recomendada: pino.
     É rápida, estruturada em JSON, e funciona bem depois com UI externa.

  2. Adicionar requestId em todas as requests
     Middleware Express:
      - gera requestId
      - guarda em req.requestId
      - devolve header x-request-id
      - todos os logs da request usam o mesmo ID

     Isto permite seguir um fluxo inteiro: UI → backend → Google → OpenAI → DB.

  3. Logging por domínio
     Criar eventos explícitos por área:

     Advisor:
      - advisor.request.started
      - advisor.openai.request
      - advisor.openai.response
      - advisor.filter.applied
      - advisor.preview.generated
      - advisor.command.apply.started
      - advisor.command.apply.completed
      - advisor.command.rejected

     Calendar:
      - calendar.connection.status
      - calendar.events.fetch.started
      - calendar.events.fetch.completed
      - calendar.event.duplicate_check
      - calendar.event.insert.started
      - calendar.event.insert.completed

     Tasks:
      - task.create
      - task.update
      - task.status_changed
      - task.due_date_changed
      - task.archive
      - task.restore

     DB:
      - db.query.slow
      - db.transaction.started
      - db.transaction.committed
      - db.transaction.rolled_back

  4. Logs ricos para filtros do advisor
     Para o teu caso atual, isto é prioritário.

     Em /ai/advisor/request, logar:
      - número de tasks enviadas ao modelo
      - número de comandos retornados pelo modelo
      - quantos caíram em cada filtro
      - IDs/títulos das tasks rejeitadas
      - motivo de rejeição

     Exemplo de evento:

     {
       "event": "advisor.calendar.filter.result",
       "generated": 20,
       "kept": 7,
       "rejected": {
         "past_event": 4,
         "duplicate_title": 3,
         "invalid_calendar": 1,
         "memory": 5
       }
     }

  5. Frontend logs leves
     Criar frontend/src/logger.ts, mas sem exagerar.

     Logar:
      - advisor request enviada
      - advisor proposals recebidas
      - proposal aceite/ignorada
      - erro de API
      - alterações de calendário default
      - Google Calendar load started/completed

     Em dev, imprimir no console.
     Em produção/local, opcionalmente enviar para backend via /client-logs.

  6. Endpoint para client logs
     Criar no backend:
      - POST /api/client-logs

     Recebe logs do frontend com:
      - level
      - event
      - message
      - metadata
      - requestId se existir

     Isto ajuda a correlacionar problemas UI/backend.

  7. Guardar logs em ficheiro
     Para já, sem infraestrutura pesada:
      - logs/app.log
      - JSON Lines, uma linha por log
      - rotação diária ou por tamanho

     Biblioteca:
      - pino
      - pino/file
      - opcional: rotating-file-stream

  8. UI própria simples dentro da app
     Eu sugiro primeiro uma UI simples no próprio projeto, não um servidor externo.

     Criar view “Logs” apenas em dev/admin:
      - lista últimos logs
      - filtro por level
      - filtro por event
      - filtro por requestId
      - pesquisa por task title/id
      - botão “copiar requestId”

     Backend endpoints:
      - GET /api/logs?level=&event=&requestId=&search=&limit=
      - lê do ficheiro logs/app.log

  9. Depois, se crescer: ferramenta externa
     Se isto ficar sério, integrar com uma destas:
      - Grafana Loki + Grafana: melhor opção open-source para logs.
      - Better Stack: mais simples, hosted, boa UI.
      - Axiom: muito bom para logs estruturados e pesquisa.
      - Sentry: melhor para erros/exceptions, não tanto para logging rico geral.

     Para este projeto, eu começaria com pino + ficheiro + UI interna. Depois migrava para Loki/Axiom se precisares.

  10. Regras de segurança
     Nunca logar:

  - cookies
  - OAuth codes
  Para OpenAI:

  - logar contagens, IDs, action, modelo, duração
  - guardar prompt completo só se LOG_AI_PAYLOADS=true
